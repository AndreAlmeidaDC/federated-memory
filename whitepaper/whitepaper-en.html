Abstract

The memory problem in AI agents is not a lack of tools. It is the absence of separation between who stores, who routes and who executes context. Current solutions centralize memory in the agent, creating tool dependency, semantic contamination across domains and inevitable degradation. This paper proposes federated memory: an architecture where context belongs to the user, domains are isolated and routing is the active work of a dedicated core. v2.0 closed three gaps left open in v1.0 — context quality feedback, deterministic memory conflict resolution and write policy enforced even with no human present — and added automatic temporal validity through file inspection. v2.1 reframes the fifth principle: human approval is no longer a mandatory bottleneck but risk-proportional auditing. Capture is automatic via core hooks; classification by confidence and risk determines what gets promoted automatically and what requires human decision. The reference implementation uses Obsidian as the base, Context Packs as the minimal unit of context, Hermes as the active core and MCP as the access interface for multiple agents.

01 · The ProblemAgents forget. And when they remember, they contaminate.

Every AI tool looks brilliant in the first few minutes. It understands the request, produces relevant output, and for a moment it seems you finally have a collaborator that follows your reasoning. The problem shows up in the second session. Or the tenth. The tool forgets what was decided, mixes context from distinct projects, repeats questions that were already answered, and needs to be re-educated continuously.

This is not a bug. It is the consequence of an architectural choice: memory, when it exists, belongs to the agent. It lives inside the tool, in the tool's format, accessible only by the tool. When you switch tools, you lose the memory. When the tool updates, the memory degrades. When you use two tools in parallel, each has a different version of reality.

The natural reaction is to try to concentrate everything in one place: a "super brain", a huge context file, a centralized system the agent reads in full before responding. At first this seems to solve it. Over time, the system degrades in another way: a marketing task pulls in code conventions; an old project contaminates a new one; memories from months ago enter as if they were still valid. The agent knows too much and, precisely because of that, begins to err in more sophisticated ways.

The problem is not that agents forget. The problem is that current architecture does not distinguish between remembering and contaminating.

The field still has no established answer for this. Memory tools exist — Mem0, Zep, Letta, native systems like Claude Code and Cursor — but they all start from the same assumption: memory is an internal component of the agency system. The agent manages the memory. The agent decides what to store. The agent decides what to bring into context.

This paper argues that this assumption is the problem, not the solution. And that separating storage from routing requires a dedicated active component — not a passive agent, not a configuration layer, but a core that makes decisions about what enters context on every call.

02 · Why Current Solutions FailThe right tools for the wrong problem.

Existing solutions are technically sophisticated. They use vector embeddings, knowledge graphs, context compression, semantic retrieval. The problem is not the technology. It is the architectural assumption that governs where memory lives and who controls it.

Native tool memory

Claude Code has CLAUDE.md and a local memory system. Cursor has project rules. Windsurf has Cascade's internal memories. Each implementation works well inside the tool that created it — and fails completely outside it. If you use Claude Code in the morning and Cursor in the afternoon for the same project, the two have different, possibly conflicting memories, with no synchronization or governance mechanism. Memory belongs to the tool, not the project, not the user.

Memory-as-a-service systems

Mem0, Zep and similar systems solve portability partially: memory lives in an external service that multiple tools query. But they introduce three new problems. First, dependency on external infrastructure — the memory is now in another company's cloud. Second, absence of explicit governance — what enters is decided automatically, without human approval, and hypotheses become permanent facts. Third, absence of semantic isolation — all memory lives in a single vector space, and distinct projects contaminate each other through semantic proximity.

Temporal knowledge graphs

Graphiti (Zep) and similar systems add a temporal and relational dimension. They are technically richer, but suffer from the same governance problem: who decides what enters the graph? By what criterion does a piece of conversation become a persistent node? Automation without curation produces graphs that grow but do not necessarily improve.

The common pattern

In all existing approaches, memory is an internal component of the agency system. The agent stores, the agent routes, the agent executes. This concentration is the root of the problem, not an inevitable property of the architecture.

03 · The Architectural ErrorStorage, routing and execution do not belong in the same place.

The fundamental problem of existing approaches is the absence of separation between three distinct functions: storing context, routing context to the right agent, and executing based on that context. When all three live in the same component — the agent — any failure in one contaminates the others.

An agent that stores its own memory has a structural incentive to store too much: more context feels safer. An agent that routes its own context has no external criterion to decide what is relevant: it brings everything or uses opaque heuristics. An agent that executes based on memory it governed itself has no audit mechanism: memory errors become silent execution errors.

An agent that stores, routes and executes with its own memory cannot know what it does not know. And it cannot be corrected without being restarted.

Mature software systems solved this problem decades ago with separation of concerns. Databases do not execute business logic. Routers do not store content. Application servers do not manage persistence directly. AI agent architecture has not yet internalized this principle.

Separating the three functions has an important corollary: the routing component cannot be implicit. It cannot be "whatever is left when the agent decides to read a file". It has to be explicit, active, with its own criteria, auditable. This is the role v1.0 of this paper underestimated — and that v2.0 corrects.

04 · The ProposalFederated memory and the three closed gaps.

Federated memory is an architecture where context belongs to the user, domains are semantically isolated, routing is explicit and governed, and agents are temporary consumers of context — not its owners.

  1. User sovereignty. The base belongs to the user. Agents come and go; context stays portable, human-readable and versionable. No tool should be a prerequisite for accessing one's own memory.

  2. Domain isolation. Context from distinct domains — marketing, engineering, writing, research — should not share the same semantic space. Cross-domain contamination is prevented by structure, not by instruction to the agent.

  3. Neutral consumption contract. A central contract file describes how any agent should consume the memory. Tool-specific adapters translate this contract but do not replace it.

  4. Minimum sufficient context. The agent receives the context needed for the task at hand, not the full history. Context Packs are the delivery unit: lean, task-oriented packages with an explicit scope of what to use and what to avoid.

  5. Human as last-instance auditor. Capture is automatic via Hermes hooks. Classification is the agent's responsibility (confidence + risk). Approval is risk-proportional: verified low-risk entries promote automatically with TTL; hypotheses and high risk require explicit human decision. The human is not a capture bottleneck — it is a quality filter.

v1.0 stated these five principles and stopped. Practice revealed three gaps that needed to be closed before the architecture was robust enough to be adopted by third parties.

Gap 1 — Context quality feedback

Context Packs were delivered, agents consumed them, and no one measured whether the package had been useful. Bad packs remained in use indefinitely because there was no rejection mechanism. The solution is structural, not cultural: each pack gets a Validation: field documenting how usage is logged, what the automatic review trigger is (three consecutive bad marks) and how the human marks a pack as obsolete. Without feedback, there is no improvement cycle. With structured feedback, bad packs have a finite half-life.

Gap 2 — Memory conflict resolution

When two notes state conflicting things, which wins? In v1.0 the answer was implicit ("whoever revised most recently"), and therefore non-deterministic — different agents reached different answers. v2.0 establishes the explicit rule: the most recent entry with status: approved wins. Entries with status: superseded remain in history but are ignored at runtime. Timestamp alone does not decide — status is mandatory. In the absence of status, the agent must ask the human, not guess. This structure — discrete entries with status and frontmatter, rather than a monolithic context file rewritten on every update — prevents by construction what the ACE paper calls context collapse: the degradation that occurs when full rewrites of context erase hard-won strategies. In federated memory, each decision is a discrete versioned entry; new information enters as incremental deltas via the inbox, never as a destructive rewrite. History is preserved by design.

Gap 3 — Writing with no human present

The principle of human approval works when the human is in front of the screen. The problem begins when the agent runs in automatic mode — scheduled, headless, in CI. v1.0 had no clear answer for this case. v2.0 establishes the rule as policy of the core itself, not as instruction to the agent: reading is always allowed across the entire vault; permanent writing is forbidden outside /90-inbox/, in any execution mode. The one enforcing this rule is not the agent (which can ignore it), but the active core that mediates the calls.

05 · The Active CoreHermes is no longer optional.

In v1.0, Hermes was described as a "lightweight router" — a useful but replaceable component, swappable for direct file reading. Practice showed this characterization underestimated the role. Without an active component mediating access to context, the three gaps above are impossible to close: quality feedback requires per-use logging; conflict resolution requires status inspection; write policy requires enforcement on the call path. None of these functions belong to the agent — they all belong to a dedicated core.

Hermes plays four roles in v2.0:

Context router

Classifies the task, identifies the relevant domain, selects the appropriate Context Pack, delivers the minimum necessary package. This is the role v1.0 recognized. The agent does not choose what to read — it receives what was routed.

Memory manager with feedback loop

Each use of a pack is logged in /99-archive/pack-usage.log with a classified result (useful, partial, bad). Three consecutive bad marks flag the pack for review. The improvement cycle is mechanical, not dependent on user habit.

Scope controller

Applies the memory conflict rule when assembling context: ignores entries with status: superseded, prioritizes the most recent with status: approved, flags the absence of status as a block. The agent never sees the conflict — it sees only the winning version or a question for the human.

Declarative policy engine

Write policy is configured once in Hermes — not in each adapter, not in each MCP call. Reading: open. Permanent writing: forbidden outside /90-inbox/. Any request outside the scope is converted into a suggestion for the inbox, even in headless execution. The policy is declarative and auditable: a single file describes what is and is not allowed.

Derived principle

Federated memory without an active core is just an organized folder. The core is what turns a file structure into an architecture: it enforces policy, measures quality and resolves conflict. Without it, the principles become good intentions.

Multimodal support is accommodated by the /assets/ structure per domain and project. The ability to process images, audio and PDFs depends on the LLM configured in the agent — not on the agent itself. Check the provider documentation for current capabilities.

06 · The DifferentiatorAutomatic temporal validity through file inspection.

Centralizing systems treat all memory content as valid until a human explicitly marks it obsolete. In practice, this means outdated content stays in circulation for months or years — no one reviews what is not visibly broken.

Federated memory has a structural advantage over centralizing systems on this point: content lives as files in the user's operating system. Files have an mtime. When delivering a Context Pack, Hermes inspects the mtime of each file listed under Use:. If any file has gone more than ninety days without an update, the pack is delivered with a warning attached to the output.

The choice to warn — and not block — is deliberate. Old content is not necessarily wrong. Engineering principles may have been written a year ago and still hold. But old content deserves a moment of checking before being used as the basis for a new decision. The warning serves that purpose without becoming friction.

Why this is a differentiator

Centralizing systems have no equivalent. Memory lives as opaque objects inside the application, with no exposed mtime, no inspection possible by an external component. Federated memory inherits this capability from the file system — it was not invented, it was recognized.

When the warning becomes action

The default behavior is informative. The human can escalate by configuring Hermes to block packs with content older than a threshold (for example, six months). For packs from volatile domains (regulatory, ML, market), the threshold can be lower; for stable domains (principles, fundamentals), higher. Configuration is per pack, not global — because aging speed is content-specific.

07 · Honest ComparisonWhere each approach wins — and where federated loses.

Federated memory is not superior in every scenario. An honest comparison requires acknowledging where other approaches have the advantage.

Where native tool memory wins

For users with a single tool, a single project and low need for portability, native memory is the right choice. Zero setup friction, perfect integration with the tool's flow, no additional infrastructure. The cost — tool dependency, lack of portability — only appears when context grows beyond what the tool can hold.

Where systems like Mem0 and Zep win

For teams that need shared memory across multiple users, with automatic semantic retrieval and no manual curation, Mem0 and Zep are a better fit. Federated requires curation discipline that does not scale automatically to large teams without additional processes.

Where centralizing systems / Life OS win

There is a category of projects that takes the opposite approach to federated: concentrating memory, automation, integrations and agents in a single monolithic application — often called "Personal AI", "Life OS" or "productivity super-app". They win on surface usability: ready-made UI, packaged integrations, visible community, near-zero adoption curve. The user installs, connects services and has something working in minutes.

The cost shows up later. Memory lives inside the application, in the application's format. Switching tools means starting from scratch. The architecture is, by design, centralizing — exactly the anti-pattern that motivated this paper. For those who prioritize adoption speed over sovereignty, it is the right choice; for those who prioritize portability and governance, it is a problem dressed up as a solution.

Where federated memory wins

For users who work with multiple agents in parallel, multiple work domains, and who need real portability — where memory must survive a tool switch — federated is the only approach that solves the problem structurally. It is also the only one that returns sovereignty to the user: memory is human-readable, versionable in Git, and requires no specific tool to be accessed.

DecisionNode: independent convergence

The DecisionNode project1, developed independently, reached partially overlapping conclusions: decisions as a structured unit, access via MCP by multiple agents, auditable history, an implicit blackboard pattern. It specifically solves the decision sub-module with semantic search via embeddings — a concrete implementation of part of the principles proposed here. The independent convergence of distinct projects toward similar conclusions is evidence that the problem is real and the architectural direction is valid.

As multiple specialized agents coexist in the same ecosystem, the blackboard pattern evolves into what we call the hive mind: each agent keeps its private memory, but reusable knowledge is published to a shared area of the vault that any agent can consult. Federated reading, controlled writing. No agent directly alters another's memory. Improvements are proposed, not imposed.

ACE: academic validation of the pattern

The Agentic Context Engineering (ACE) paper, published in 2025 by researchers from Stanford, SambaNova Systems and UC Berkeley (arXiv:2510.04618), arrived independently at a formulation that validates the architecture proposed here. ACE treats context as a "playbook" that evolves over time, maintained by three roles: Generator (executes the task and produces the trajectory), Reflector (analyzes successes and errors and extracts lessons) and Curator (consolidates the lessons into incremental updates of the playbook). The parallel with the federated architecture is direct: the executing agent (Claude Code, Cursor, etc.) is the Generator; capture via hooks with confidence + risk classification is the Reflector; the review-inbox with promote-skills and the approved/superseded rule is the Curator; and the federated vault is the playbook. The difference is philosophical and is the differentiator of this architecture: in ACE, the three roles are fully automatic, executed by LLMs. In federated memory, the Curator keeps the human as the last-instance auditor. Low risk consolidates automatically; hypotheses and high risk require human decision. In one phrase: this architecture is ACE with risk-proportional governance. ACE reports gains of +10.6% on agent tasks and +8.6% on domain reasoning with no fine-tuning — evidence that the direction (context as an evolving playbook, not a static prompt) is technically sound.

Complementary layers: memory and orchestration

Systems like Paperclip (paperclip.ing) solve an adjacent but distinct problem. They are not memory systems — they are agent-team orchestration platforms, with org charts, per-agent budgets, scheduled heartbeats and hierarchical governance. The distinction matters: federated memory is context infrastructure per agent. Orchestration is coordination infrastructure between agents. Whoever uses both layers together has the most complete solution: persistent and governed context on one side, structured coordination on the other. Paperclip even uses its own SKILLS.md for runtime context injection — convergence with the /50-skills/ folder of this architecture. They are complementary projects, not competitors.

At the other end of the spectrum, Pi (pi.dev) is minimalist by design and has no native MCP. It has its own AGENTS.md, skills and automatic compaction. Integration with the federated vault happens via direct filesystem — running Pi inside the vault already loads the AGENTS.md automatically, with no need to configure an MCP server. Federated memory adds what Pi does not offer on its own: portability across machines, domain isolation and governance with human approval.

08 · Known LimitationsWhat this architecture does not yet solve.

Adopting federated memory does not buy you every problem solved. Five limitations deserve to be acknowledged explicitly — not to justify the architecture, but to make clear where it ends and where the user needs to fill the gap.

1. Action rollback does not exist

The core enforces policy over what enters context and what can be written. It does not undo the action the agent already executed. If an agent, upon receiving a Context Pack, produces bad code, sends a wrong email or applies a change to an external system, the federated architecture has no way to revert it. Rollback remains a problem of the target system (Git for code, sent folder for email, transactions for databases). Federated memory improves the chance the agent does the right thing; it does not guarantee the action is reversible when it fails.

This limitation is partially mitigated by pre-action logging: before executing high-risk actions, the agent records in /99-archive/pre-action-log.md the active context, the loaded pack, the executing agent and the target resource. It is not rollback — it is traceability. If something goes wrong, you know exactly what was active when the error happened. For reversible actions like code, Git covers rollback; the pre-action log covers the audit.

2. Temporal validity is a warning, not a guarantee

Automatic mtime inspection warns when Context Pack files have passed the configured age threshold. The warning is informative, not blocking by default. Old content may still be valid; freshly updated content may be wrong. This limitation is mitigated by the review_date field in Context Packs and Decisions: beyond mtime, Hermes checks whether next_review has expired — independent of whether the file was modified. The human who reviews the content and confirms it is still valid updates review_date without altering the file. The system does not replace content review — it makes the point where review is needed more visible.

3. Governance depends on human discipline

With automatic classification by confidence + risk, the inbox accumulates only hypotheses and high-risk decisions. Everything that is verified + low promotes automatically in 7 days. The human does not need to review what is routine — only what is uncertain or consequential. The real limitation is more precise: if you do not review the inbox, hypotheses and high-risk decisions do not become memory. The automatic flow keeps operating for the rest. The capture hooks (sync_turn, prefetch, shutdown) reduce dependence on manual action even further.

4. Scale has not been tested — and need not be a ceiling

The reference implementation was exercised at small scale (one person, few domains, a handful of projects). Behavior at medium or large scale is architectural projection, not field data. But scale is not a ceiling — it is a documented progression with concrete steps:

  • Small scale — single vault, Markdown, Hermes as core, MCP as interface. Works well for one person or a small team with well-separated domains.
  • Medium scale — when temporal questions appear ("what changed?", "which decision is most recent?", "who decided what?"), Graphiti enters as a layer of relations and history over the existing vault. It does not replace Obsidian — it complements it, indexing what the Markdown already contains.
  • Large scale — multiple physical vaults per client, team or project, each with its own Hermes and MCP. Graphiti becomes the only component with a cross-vault view, indexing decisions and relations that cross isolation boundaries.
The scale limitation is not an architectural ceiling. It is a deliberate entry point. Graphiti enters when the vault needs temporal memory. Multiple vaults enter when physical isolation becomes necessary. The architecture grows with the pain, not before it.

5. Hive mind: structure implemented, scale is v3.0

The hive mind structure is implemented: /50-skills/published/, /proposed/, /deprecated/, INDEX.md and the publication protocol via inbox. The promote-skills.mjs and update-index.mjs scripts automate TTL promotion and index updates. What is still v3.0: integration with Graphiti for relations between skills, semantic search in the index, and a local worker that indexes automatically once the ecosystem of specialized agents grows enough for sharing to make operational sense.

09 · Principles and LimitsWhat this architecture refuses.

The AI agent field matured quickly on the capability layer — reasoning, code generation, document analysis. The memory layer fell behind, and the gap is becoming the main limiter of continuous use. It is not a lack of agent intelligence. It is a lack of context continuity across sessions, projects and tools.

Current solutions treat memory as a product feature: something each tool implements internally, in a proprietary way, with the user as a passive beneficiary. This paper's proposal inverts that relationship: memory is the user's infrastructure, not the product's. Agents are temporary consumers of a context that belongs to whoever does the work, not to whoever provides the tool.

Adopting this architecture does not depend on changes to AI agents. It depends on a shift of assumption about where memory lives. When memory belongs to the user — readable, versionable, portable, governed — switching tools stops being a loss.

When memory belongs to the user, the tool becomes disposable. This is, precisely, what the AI agent market needs in order to mature.

This architecture refuses four paths. It refuses a hosted backend as a service — it breaks sovereignty. It refuses automatic memory without review — it is the central anti-pattern. It refuses a single universal adapter — each agent has its own conventions, and a generic adapter becomes the worst common denominator. It refuses RAG on the critical path — embeddings help with search, but never replace Markdown as the source.

What this architecture accepts: being slower to adopt than a monolithic application; requiring curation discipline that automatic tools do not; asking the user to learn the contract before delegating. The price of sovereignty is active participation.
