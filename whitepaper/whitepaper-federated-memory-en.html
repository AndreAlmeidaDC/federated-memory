Abstract

The memory problem in AI agents is not a lack of tools. It is the absence of separation between who stores, who routes, and who executes context. Current solutions centralize memory in the agent, creating tool dependency, semantic contamination across domains, and inevitable degradation over time. This paper proposes federated memory: an architecture where context belongs to the user, domains are isolated, routing is explicit, and human approval governs what becomes permanent truth. The reference implementation uses Obsidian as a curated storage base, Context Packs as the minimum unit of context, Hermes as a router, and MCP as a standardized access interface for multiple agents.

01 · The ProblemAgents forget. And when they remember, they contaminate.

Every AI tool seems brilliant in the first few minutes. It understands the request, produces relevant output, and for a moment it feels like you finally have a collaborator that follows your reasoning. The problem appears in the second session. Or the tenth. The tool forgets what was decided, mixes context from different projects, repeats questions that were already answered, and needs to be re-educated continuously.

This is not a bug. It is a consequence of an architectural choice: memory, when it exists, belongs to the agent. It lives inside the tool, in the tool's format, accessible only through the tool. When you switch tools, you lose the memory. When the tool updates, the memory degrades. When you use two tools in parallel, each one holds a different version of reality.

The natural reaction is to try to consolidate everything in one place: a "super brain," a massive context file, a centralized memory system that the agent reads in full before responding. At first, this seems to work. Over time, the system degrades in a different way: a marketing task pulls in coding conventions; an old project contaminates a new one; memories from months ago enter as if they were still valid. The agent knows too much and, precisely because of that, starts making more sophisticated mistakes.

The problem is not that agents forget. The problem is that the current architecture does not distinguish between remembering and contaminating.

The field still lacks an established answer to this. Memory tools exist — Mem0, Zep, Letta, native memory systems in tools like Claude Code and Cursor — but all share the same assumption: memory is an internal component of the agent system. The agent manages memory. The agent decides what to store. The agent decides what to bring into context.

This paper argues that this assumption is the problem, not the solution.

02 · Why Current Solutions FailThe right tools for the wrong problem.

Existing memory solutions for AI agents are technically sophisticated. They use vector embeddings, knowledge graphs, context compression, semantic retrieval. The problem is not the technology. It is the architectural assumption that governs where memory lives and who controls it.

Native tool memory

Claude Code has CLAUDE.md and a local memory system. Cursor has project rules. Windsurf has per-workspace memory. Each of these implementations works well inside the tool that created it — and fails completely outside of it. If you use Claude Code in the morning and Cursor in the afternoon for the same project, both tools hold different, possibly conflicting memories, with no synchronization or governance mechanism. The memory belongs to the tool, not to the project, not to the user.

Memory-as-a-service systems

Mem0, Zep, and similar services partially solve the portability problem: memory lives in an external service that multiple tools can query. But they introduce three new problems. First, dependency on external infrastructure — the memory now lives on another company's cloud. Second, absent explicit governance — what enters memory is decided automatically, without human approval, and hypotheses become permanent facts without review. Third, absent semantic isolation — all memory lives in a single vector space, and distinct projects contaminate each other through semantic proximity.

Temporal knowledge graphs

Graphiti (Zep) and similar systems add temporal and relational dimensions to memory. They are technically richer, but suffer from the same governance problem: who decides what enters the graph? By what criterion does information from a conversation become a persistent node? Automation without human curation produces graphs that grow, but do not necessarily improve.

The common pattern

In all existing approaches, memory is an internal component of the agent system. The agent stores, the agent routes, the agent executes. This concentration is the root of the problem, not an inevitable property of the architecture.

03 · The Architectural MistakeMixing storage, routing, and execution.

The fundamental problem with existing approaches is the absence of separation between three distinct functions: storing context, routing context to the right agent, and executing based on that context. When these three functions live in the same component — the agent — any failure in one contaminates the others.

An agent that stores its own memory has a structural incentive to over-store: more context feels safer. An agent that routes its own context has no external criterion to decide what is relevant: it brings everything or uses opaque heuristics. An agent that executes based on memory it itself governed has no audit mechanism: memory errors silently become execution errors.

The solution is not a smarter agent. It is an architecture that separates these responsibilities.

An agent that stores, routes, and executes with its own memory cannot know what it does not know. And it cannot be corrected without being restarted.

Mature software systems solved this problem decades ago with the principle of separation of concerns. Databases do not execute business logic. Routers do not store content. Application servers do not manage persistence directly. AI agent architecture has not yet internalized this principle.

The important corollary: when storage, routing, and execution are separated, replacing any one of the three does not destroy the system. You can swap the executor agent without losing memory. You can change the router without rewriting the access contract. You can migrate storage without the agents noticing. That is what real portability means.

04 · The ProposalFederated memory: principles before tools.

Federated memory is an architecture where context belongs to the user, domains are semantically isolated, routing is explicit and governed, and agents are temporary consumers of context — not owners of it.

The proposal has five principles that remain valid regardless of the tools chosen to implement them:

  1. User sovereignty. The memory base belongs to the user. Agents come and go; context remains portable, human-readable, and versionable. No tool should be a prerequisite for accessing one's own memory.

  2. Domain isolation. Contexts from distinct domains — marketing, engineering, writing, research — must not share the same semantic space. Cross-domain contamination is prevented by structure, not by instructions to the agent.

  3. Neutral consumption contract. A central contract file describes how any agent should consume memory. Tool-specific adapters translate this contract but do not replace it. Switching tools means switching the adapter, not rewriting the rules.

  4. Minimum sufficient context. The agent receives the context needed for the task at hand, not the full memory history. Context Packs are the delivery unit: lean, task-oriented packages with explicit scope of what to use and what to avoid.

  5. Human approval for permanent memory. No information from a conversation, hypothesis, or draft becomes permanent memory without explicit human review. Automation may suggest; the human approves. This distinction is what prevents the inevitable degradation of automatic memory systems.

Coordination without direct communication

A central insight of the federated architecture is that multiple agents do not need to communicate with each other in order to be coordinated. Coordination happens through shared memory — what distributed systems literature calls a blackboard architecture.

Claude Code accesses the memory base in the morning, produces a result, and suggests a memory update. The human approves. In the afternoon, Cursor accesses the same updated base and works with the latest state — without the two agents ever having exchanged a message. The shared board is the coordination. This eliminates the need for inter-agent communication protocols, message queues, and the inherent complexity of tightly coupled multi-agent systems.

Practical implication

Replacing one agent with another does not break coordination. The new agent reads the same board. Memory does not need to be migrated, re-imported, or reconfigured. Portability is a structural consequence of the architecture, not a feature added later.

The minimum unit: the Context Pack

Context Packs are the most underestimated piece of the architecture. They are short files that define, for a specific task: the objective, which sources in the memory base to use, which sources to explicitly avoid, the expected output format, and the temporal validity of the context.

The "avoid" list is just as important as the "use" list. A Context Pack for LinkedIn writing that explicitly avoids engineering notes does not rely on the agent knowing that engineering notes are irrelevant — that is encoded in the pack. Semantic isolation is structural, not instructional.

05 · Comparison with AlternativesWhere each approach wins and where it loses.

Federated memory is not superior in every scenario. An honest comparison requires acknowledging where other approaches have the advantage.

Where native tool memory wins

For users with a single tool, a single project, and low portability needs, native memory is the right choice. It has zero setup friction, perfect integration with the tool's workflow, and requires no additional infrastructure. The cost — tool dependency, absence of portability — only appears when context grows beyond what the tool can handle.

Where systems like Mem0 and Zep win

For teams that need shared memory across multiple users, with automatic semantic retrieval and without manual curation, Mem0 and Zep are more suitable. The federated memory proposed here requires curation discipline that does not scale automatically to large teams without additional governance processes.

Where federated memory wins

For users who work with multiple agents in parallel, across multiple work domains, and who need real portability — where memory must survive tool switching — the federated architecture is the only approach that solves the problem structurally. It is also the only one that returns sovereignty to the user: memory is human-readable, versionable in Git, and requires no specific tool to be accessed.

DecisionNode: independent convergence

The DecisionNode project1, developed independently, reached partially overlapping conclusions: decisions as structured units, MCP access by multiple agents, auditable history, implicit blackboard pattern. It specifically solves the decisions sub-module with semantic search via embeddings — a concrete implementation of part of the principles proposed here. The independent convergence of distinct projects toward similar conclusions is evidence that the problem is real and the architectural direction is valid.

06 · Reference ImplementationObsidian + Hermes + MCP.

The principles from the previous section are tool-agnostic. The reference implementation described here uses specific choices that can be replaced as long as the principles are preserved.

Obsidian as the storage base: human-readable Markdown files, versionable in Git, with support for links, backlinks, and navigation. Replaceable by any system that preserves human readability and versioning.

Hermes Agent (NousResearch) as the router: classifies the task, identifies the domain, selects the relevant Context Pack, and delivers context in the format expected by the executor agent. Its native ability to read context files like AGENTS.md allows direct integration with the memory base without additional code.

MCP server for Obsidian as the access interface: exposes the vault in a standardized way to any MCP client — Claude Code, Cursor, Windsurf, Antigravity — with granular read and write permission control.

Context Packs as the delivery unit: short per-task files that define scope, sources, constraints, and expected format. They are the piece that transforms a memory base into usable context.

Graphiti (Zep) as an optional upgrade: enters only when memory needs a temporal dimension — decisions that supersede decisions, relationships between entities, change history. It does not replace Obsidian; it complements with capabilities that plain Markdown does not offer.

Substitution criterion

Any component can be replaced. The criterion is not the tool — it is the principle the tool implements. Obsidian can be Logseq. Hermes can be LangGraph. MCP can be a local CLI. What cannot change: user sovereignty, domain isolation, neutral contract, minimum context, human approval.

07 · ConclusionMemory is infrastructure. Not a product feature.

The AI agent field has matured rapidly at the capability layer — reasoning, code generation, document analysis. The memory layer has fallen behind, and the gap is becoming the primary limitation for sustained use. It is not a lack of agent intelligence. It is the lack of context continuity across sessions, projects, and tools.

Current solutions treat memory as a product feature: something each tool implements internally, in a proprietary way, with the user as a passive beneficiary. This paper's proposal inverts that relationship: memory is the user's infrastructure, not the product's. Agents are temporary consumers of a context that belongs to whoever does the work, not to whoever provides the tool.

Adopting this architecture requires no changes to AI agents. It requires a change in the assumption about where memory lives. When memory belongs to the user — readable, versionable, portable, governed — switching tools no longer means losing context. Improving an agent no longer depends on a company's goodwill. And degradation from poorly managed memory ceases to be inevitable.

When memory belongs to the user, the tool becomes disposable. That is exactly what the AI agent market needs in order to mature.

Tools like DecisionNode, developed independently and arriving at similar conclusions, are evidence that the market is converging in this direction. This paper attempts to name the pattern before it becomes merely a product feature of a specific company — and to establish the principles that any implementation must preserve for user sovereignty to be real, not merely declared.
